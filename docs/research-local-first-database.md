# Research: Local-First Sync Options for Grimoire (Plain Postgres vs Zero vs ElectricSQL)

> Researched 2026-07-05 against primary sources only (official docs, first-party blogs, GitHub). Every claim links to its source. Where docs are silent or ambiguous, that is stated explicitly.

Grimoire is a Laravel 13 + React 19 + Inertia app planned for Laravel Cloud, whose managed "Serverless Postgres" is [powered by Neon](https://cloud.laravel.com/docs/resources/databases/postgres). This document evaluates three paths: (a) plain managed Postgres via Laravel Cloud, (b) [Zero by Rocicorp](https://zero.rocicorp.dev/), and (c) [ElectricSQL](https://electric-sql.com/).

## TL;DR comparison

| | Plain Postgres (Laravel Cloud) | Zero (Rocicorp) | ElectricSQL |
|---|---|---|---|
| **Extra infra required** | None | `zero-cache` (Node, Docker image `rocicorp/zero`) with persistent disk for a SQLite replica, WebSocket-capable load balancer; ports 4848/4849 ([self-host docs](https://zero.rocicorp.dev/docs/self-host)). Managed option: [Cloud Zero](https://zero.rocicorp.dev/docs/cloud-zero), from $30/mo ([pricing](https://zero.rocicorp.dev/)) | Electric sync service (Elixir, Docker image `electricsql/electric`) with persistent filesystem for shape logs, HTTP port; ideally behind a CDN ([deployment guide](https://electric-sql.com/docs/guides/deployment)). Managed option: Electric Cloud ([deployment guide](https://electric-sql.com/docs/guides/deployment)) |
| **Works with Laravel Cloud managed Postgres?** | ✅ Yes, that's the product | ⚠️ **Unverified / likely blocked.** Zero requires `wal_level=logical` + a direct replication connection ([Zero Postgres docs](https://zero.rocicorp.dev/docs/connecting-to-postgres)). Laravel Cloud's docs and API expose no way to enable logical replication and no Neon console access (see [§2](#2-laravel-cloud-managed-postgres)) | ⚠️ Same blocker: Electric requires logical replication + a `REPLICATION`-attribute role ([deployment guide](https://electric-sql.com/docs/guides/deployment)); not documented as possible on Laravel Cloud |
| **Works with direct-provisioned Neon?** | ✅ ([Neon Laravel guide](https://neon.com/docs/guides/laravel)) | ✅ Neon listed as fully supported, with caveats about always-on compute billing ([Zero Postgres docs](https://zero.rocicorp.dev/docs/connecting-to-postgres), [Neon's Zero guide](https://neon.com/guides/zero)) | ✅ First-party integration docs on both sides ([Electric's Neon page](https://electric-sql.com/docs/integrations/neon), [Neon's Electric guide](https://neon.com/guides/electric-sql)) |
| **Write path** | Normal Laravel controllers/Eloquent | Custom mutators: optimistic on client, then pushed to *your* API's mutate endpoint, which writes to Postgres. Endpoint "can be implemented in any programming language" (protocol not yet formally documented) ([mutators docs](https://zero.rocicorp.dev/docs/mutators)) | Read-path only by design; writes go through your existing API (e.g. Laravel controllers). Four documented patterns from plain online writes to through-the-DB sync ([writes guide](https://electric-sql.com/docs/guides/writes)) |
| **Offline support** | None (server-rendered) | Local store (IndexedDB) + optimistic mutations queued and replayed ([auth docs](https://zero.rocicorp.dev/docs/auth), [mutators docs](https://zero.rocicorp.dev/docs/mutators)) | Offline reads out of the box; offline *writes* are your responsibility via optimistic-state patterns ([writes guide](https://electric-sql.com/docs/guides/writes)) |
| **Maturity / license** | GA product; Neon under the hood | GA since 1.0 (March 2026), currently 1.7; Apache-2.0 ([status](https://zero.rocicorp.dev/docs/status), [release notes](https://zero.rocicorp.dev/docs/release-notes), [license](https://zero.rocicorp.dev/docs/open-source)) | GA since 1.0 (March 2025); Apache-2.0 ([1.0 announcement](https://electric-sql.com/blog/2025/03/17/electricsql-1.0-released), [GitHub](https://github.com/electric-sql/electric)) |
| **Operational complexity** | Lowest | Highest: stateful sync service + replication slot + SQLite replica lifecycle + query/mutate endpoints + client schema in TS | Medium: stateless-ish HTTP service + replication slot + shape storage; writes reuse your existing Laravel API |

**Bottom line:** Both sync engines are database sync engines whose source of truth is *your* Postgres — neither gives users Obsidian-style files they own on disk (see [§7](#7-the-local-files-angle)). Both need `wal_level=logical`, which Laravel Cloud does not document any way to enable on its Neon-backed databases, so either engine realistically means provisioning Neon (or other Postgres) directly and running one extra always-on service.

## 1. Zero's requirements on Postgres

Source: [Connecting to Postgres](https://zero.rocicorp.dev/docs/connecting-to-postgres) (Zero docs).

- **Postgres version & replication.** "Zero requires Postgres v15.0 or higher, and support for logical replication." The `wal_level` config parameter "has to be set to `logical`" ([docs](https://zero.rocicorp.dev/docs/connecting-to-postgres)).
- **Event triggers (optional but important).** Zero uses Postgres event triggers "when possible to implement high-quality, efficient schema migration." On providers without event trigger access, Zero still works, "but for correctness, any schema change triggers a full reset of all server-side and client-side state. For small databases (< 10GB) this can be OK" ([docs](https://zero.rocicorp.dev/docs/connecting-to-postgres)). For such providers Zero offers manual [schema change hooks](https://zero.rocicorp.dev/docs/connecting-to-postgres) (`zero_0.update_schemas()` called inside the DDL transaction).
- **Permissions.** The docs don't demand superuser universally, but provider notes show what's needed: on Fly.io, "Fly Managed Postgres does not provide superuser access, so `zero-cache` cannot create event triggers," and publication creation (`FOR ALL TABLES`) can be permission-restricted, requiring a manually created publication ([docs](https://zero.rocicorp.dev/docs/connecting-to-postgres)). So in practice zero-cache needs: a role that can create replication slots and (ideally) publications and event triggers.
- **Connection type.** The upstream connection (`ZERO_UPSTREAM_DB`) "*must* be a direct connection (not via pgbouncer)"; pooled URLs are recommended for the auxiliary `ZERO_CVR_DB` and `ZERO_CHANGE_DB` databases ([self-host docs](https://zero.rocicorp.dev/docs/self-host), [Postgres docs — PlanetScale pooling note](https://zero.rocicorp.dev/docs/connecting-to-postgres)).

### Zero + Neon specifically

Neon is listed as **"✅ See notes below"** in Zero's provider support table ([docs](https://zero.rocicorp.dev/docs/connecting-to-postgres)). The Neon-specific notes:

- "Neon supports logical replication, but you need to enable it in the Neon console for your branch/endpoint" ([Zero docs, Neon section](https://zero.rocicorp.dev/docs/connecting-to-postgres)).
- **Billing caveat:** "because Zero keeps an open connection to Postgres to replicate changes, as long as zero-cache is running, Postgres will be running and you will be charged by Neon. … for smaller applications that would otherwise not always be running, this can create a surprisingly high bill" ([Zero docs](https://zero.rocicorp.dev/docs/connecting-to-postgres)). Neon's own docs confirm: a connected logical replication subscriber prevents scale-to-zero ([Neon scale-to-zero docs](https://neon.com/docs/guides/scale-to-zero-guide)).
- **Branching caveat:** Neon preview branches can each end up kept always-on by Zero "if not done with care" ([Zero docs](https://zero.rocicorp.dev/docs/connecting-to-postgres)).
- Neon's first-party guide for Zero reiterates: enable logical replication in Console → Settings → Logical Replication, and use the **direct (non-pooled) connection string** for `ZERO_UPSTREAM_DB` because "the pooler does not support the required replication protocol" ([Neon guide: Getting started with Zero](https://neon.com/guides/zero)).
- Note on the question about "custom endpoint settings": current Zero docs describe enabling logical replication via the Neon **console setting**, not via custom endpoint config JSON. Neon's docs note enabling it "permanently changes `wal_level` to `logical`," cannot be reverted, and restarts all computes; also `max_wal_senders` and `max_replication_slots` are both fixed at 10, and **inactive replication slots are removed after ~40 hours** ([Neon logical replication docs](https://neon.com/docs/guides/logical-replication-neon)) — that last point matters if zero-cache is ever offline for a couple of days.
- Zero's docs don't state whether Neon roles can create event triggers. Neon documentation for another tool (pgstream) indicates only the `neondb_owner` role has event-trigger privileges ([Xata pgstream Neon docs](https://xata.io/docs/opensource/pgstream/docs/neon) — third-party, flagged as such). **Unverified for Zero specifically.**

## 2. Laravel Cloud managed Postgres

Source: [Serverless Postgres — Laravel Cloud docs](https://cloud.laravel.com/docs/resources/databases/postgres).

- **What it is.** "Fully managed, autoscaling Serverless Postgres database. Powered by Neon." Compute autoscales between configurable compute-unit bounds (0.25–4 CU on the docs page; the API schema allows up to 10), storage 5–1,000 GB, and scale-to-zero after a configurable idle period ([docs](https://cloud.laravel.com/docs/resources/databases/postgres), [update-cluster API schema](https://cloud.laravel.com/docs/api/database-clusters/update-database-cluster.md)).
- **Direct connection string: yes.** You can view full credentials (host, username, password) via "Org → Resources → Databases → … → View credentials" for connecting "from your local machine" with tools like TablePlus ([docs](https://cloud.laravel.com/docs/resources/databases/postgres)). Hostnames look like `ep-xxx.us-east-2.pg.laravel.cloud` and SSL uses the standard ISRG Root X1 CA ([docs](https://cloud.laravel.com/docs/resources/databases/postgres)). Nothing in the docs says external services are prohibited from using these credentials, but the docs only ever describe app and local-machine access — **whether an external always-on service (like zero-cache) is a supported use is not documented**.
- **PgBouncer pooler: yes.** Append `-pooler` to the first hostname segment for a pgbouncer endpoint supporting up to 10,000 connections ([docs](https://cloud.laravel.com/docs/resources/databases/postgres)).
- **Custom Neon endpoint settings / `wal_level=logical`: not exposed.** The docs describe only cluster name, instance size (compute units), storage, and region as configurable ([docs](https://cloud.laravel.com/docs/resources/databases/postgres)). The public API's `UpdateDatabaseRequest` for `neon_serverless_postgres_*` types accepts exactly four fields: `cu_min`, `cu_max`, `suspend_seconds`, `retention_days` ([API schema](https://cloud.laravel.com/docs/api/database-clusters/update-database-cluster.md)). There is **no documented setting for logical replication, `wal_level`, publications, replication roles, or Neon console access**, and no page about logical replication exists anywhere in the docs index ([docs index](https://cloud.laravel.com/docs/llms.txt)). Since Neon's own procedure requires flipping a project-level setting in the Neon console or API ([Neon docs](https://neon.com/docs/guides/logical-replication-neon)), and Laravel Cloud does not document giving you that access, **running Zero or Electric against Laravel Cloud Serverless Postgres appears not currently possible — though this is an absence of documentation, not an explicit "no." Confirming with Laravel Cloud support would be the definitive check.**
- Laravel Cloud also offers AWS RDS Postgres, but "available exclusively through Laravel Private Cloud" ([laravel.com/cloud/app-resources](https://laravel.com/cloud/app-resources)); Private Cloud RDS docs exist ([RDS docs](https://cloud.laravel.com/docs/private-cloud/rds.md)) but whether logical replication parameters are configurable there is likewise not documented.

## 3. Deploying zero-cache

Source: [Self-Hosting Zero](https://zero.rocicorp.dev/docs/self-host).

- **What it is.** zero-cache consists of one **replication-manager** (owns the Postgres replication slot) and N horizontally scalable **view-syncers** (serve client queries over WebSockets from a SQLite replica). "Zero-cache is backed by a SQLite replica of your database" ([docs](https://zero.rocicorp.dev/docs/self-host)).
- **Runtime requirements.** Shipped as Docker image `rocicorp/zero` (a Node service — the docs note "Node is single-threaded" when discussing sync workers). It needs: a persistent volume for `ZERO_REPLICA_FILE` (the SQLite replica; the docs stress optimizing disk IOPS for it), env vars for `ZERO_UPSTREAM_DB` / `ZERO_CVR_DB` / `ZERO_CHANGE_DB`, and `ZERO_QUERY_URL` / `ZERO_MUTATE_URL` pointing at your API. View-syncers "must be publicly reachable by clients on port 4848" behind a WebSocket-capable load balancer (sticky sessions recommended); the replication-manager listens on 4849 and must stay private. Multi-node setups back up the replica to S3 via `ZERO_LITESTREAM_BACKUP_URL` ([docs](https://zero.rocicorp.dev/docs/self-host)).
- **Recommended deployment targets.** The docs give worked configs for Docker Compose, Fly.io, SST/AWS (ECS + EFS), and Kubernetes ([docs](https://zero.rocicorp.dev/docs/self-host)). For most users Rocicorp recommends the managed [Cloud Zero](https://zero.rocicorp.dev/docs/cloud-zero) service instead (Standard $30/mo with 10 GB storage; Pro $300/mo; BYOC from $1,000/mo — [zero.rocicorp.dev pricing](https://zero.rocicorp.dev/)).
- **Can it run on Laravel Cloud?** Effectively no. Laravel Cloud compute runs your Laravel/Symfony application (PHP 8.2–8.5 runtimes only — [runtimes docs](https://cloud.laravel.com/docs/runtimes.md)); worker clusters are "a mirror copy of your environment's App compute … same repository, branch, PHP version" for queue workers and background processes ([compute docs](https://cloud.laravel.com/docs/compute.md)). There is no documented way to run an arbitrary long-lived Node/Docker service with a persistent volume and its own public WebSocket port. zero-cache would live on separate hosting (Fly.io, AWS, Render, a VPS) or Cloud Zero. Note the API behind the query/mutate URLs "can live anywhere zero-cache can reach" ([self-host docs](https://zero.rocicorp.dev/docs/self-host)) — so the Laravel app can stay on Laravel Cloud.

## 4. Zero's write path, auth, and permissions

Sources: [Mutators](https://zero.rocicorp.dev/docs/mutators), [Queries](https://zero.rocicorp.dev/docs/queries), [Authentication](https://zero.rocicorp.dev/docs/auth).

- **Writes = custom mutators.** A mutator runs optimistically on the client against the local store; Zero then sends the mutation (name + args) to *your server's* push endpoint, registered with zero-cache via `ZERO_MUTATE_URL`. Your endpoint "runs the push protocol, executing the server-side mutator in a transaction against your database"; results replicate back through zero-cache and pending optimistic effects are rolled back/reconciled ([mutators docs](https://zero.rocicorp.dev/docs/mutators)). Server mutators can differ from client ones (extra permission checks, audit logs, raw SQL via `dbTransaction`) ([mutators docs](https://zero.rocicorp.dev/docs/mutators)).
- **Non-JS backends (i.e. Laravel).** The docs provide TypeScript helpers (`handleMutateRequest`, adapters for Drizzle/Kysely/Prisma/pg), but explicitly state: "You can manually implement the mutate endpoint in any programming language. This will be documented in the future, but you can refer to the handleMutateRequest source code for an example for now" ([mutators docs](https://zero.rocicorp.dev/docs/mutators)). Similarly for reads: "It is possible to implement the `ZERO_QUERY_URL` endpoint without using Zero's TypeScript libraries, or even in a different language entirely" — the query endpoint receives `{id, name, args}[]` JSON and must respond with ZQL ASTs per the [zero-protocol AST schema](https://github.com/rocicorp/mono/blob/main/packages/zero-protocol/src/ast.ts) ([queries docs](https://zero.rocicorp.dev/docs/queries)). **Caveat:** a Laravel implementation means hand-implementing an undocumented-but-stable-ish wire protocol (push protocol + AST construction) in PHP with no first-party PHP SDK; the officially supported path is TS/JS. The mutator definitions (validation, conflict-resolution copies) must also exist in TypeScript on the client regardless.
- **Auth.** Both cookies and tokens are supported. Cookie mode: set `ZERO_QUERY_FORWARD_COOKIES` / `ZERO_MUTATE_FORWARD_COOKIES` and zero-cache forwards the browser's cookies to your endpoints (requires zero-cache on a subdomain of your site and cookies scoped to the root domain). Token mode: pass a JWT (or any opaque token) to the client's `auth` option; zero-cache forwards it as `Authorization: Bearer` to your query/mutate endpoints, which validate it however you like. 401/403 responses put the client in a `needs-auth` state until it reconnects with fresh credentials ([auth docs](https://zero.rocicorp.dev/docs/auth)). For a Laravel app, this means Zero auth can ride on Laravel's existing session cookie or a Sanctum/JWT token — validation happens in your endpoint, not in zero-cache.
- **Read permissions.** "Zero does not have (or need) a first-class permission system like RLS." Clients cannot send arbitrary queries; they reference named queries you define, and zero-cache calls your query endpoint to resolve name+args into ZQL, where you enforce permissions using the authenticated context (e.g. `zql.post.where('authorID', ctx.id)`); deny by returning a query matching no rows ([auth docs — Permission Patterns](https://zero.rocicorp.dev/docs/auth), [llms.txt overview](https://zero.rocicorp.dev/llms.txt)). Write permissions are plain code inside server mutators ([mutators docs](https://zero.rocicorp.dev/docs/mutators)).

## 5. Zero maturity and licensing

- **Status: GA.** "As of March 2026, Zero is generally available and fully-supported" ([project status](https://zero.rocicorp.dev/docs/status)). 1.0 shipped as the first stable release; the version current at the time of writing is **1.7** ("Query Correctness and Performance") ([release notes](https://zero.rocicorp.dev/docs/release-notes)); the self-host docs pin `rocicorp/zero:1.7.0` ([self-host docs](https://zero.rocicorp.dev/docs/self-host)). 2026 roadmap includes Cloud Zero full public availability and column permissions; aggregates, SSR, and first-class text search are "would like to, not committing" ([status](https://zero.rocicorp.dev/docs/status)).
- **License: Apache-2.0** for both client and server ([open source page](https://zero.rocicorp.dev/docs/open-source), [LICENSE in rocicorp/mono](https://github.com/rocicorp/mono/blob/main/LICENSE)). Business model is the managed service and paid support ([open source page](https://zero.rocicorp.dev/docs/open-source)).
- **Pricing (managed only; self-host is free).** Cloud Zero: Standard $30/mo (10 GB storage, 3 shared vCPU), Pro $300/mo (100 GB, 7 dedicated vCPU, SLA), BYOC $1,000+/mo in your AWS account; extra storage $0.20/GB ([zero.rocicorp.dev](https://zero.rocicorp.dev/)). Cloud Zero is not yet at "full public availability" per the [roadmap](https://zero.rocicorp.dev/docs/status).

## 6. ElectricSQL as the alternative

Sources: [Deployment guide](https://electric-sql.com/docs/guides/deployment), [Writes guide](https://electric-sql.com/docs/guides/writes), [Neon integration](https://electric-sql.com/docs/integrations/neon), [1.0 announcement](https://electric-sql.com/blog/2025/03/17/electricsql-1.0-released).

- **Postgres requirements.** "Any standard Postgres, version 14 and above … Postgres must have logical replication enabled. You also need to connect as a database role that has the `REPLICATION` attribute." Direct connection required (not a pooler) for replication; an optional `ELECTRIC_POOLED_DATABASE_URL` can use a pooler for other queries. Electric creates a publication (`electric_publication_default`) and replication slot (`electric_slot_default`); if its role can't create the publication, tables must be added to it by hand ([deployment guide](https://electric-sql.com/docs/guides/deployment)). Notably, Electric does **not** need event triggers (unlike Zero's optimal path) and works with "any Postgres data model" ([deployment guide](https://electric-sql.com/docs/guides/deployment)).
- **Deployment model.** "The Electric sync engine is an Elixir web service, packaged using Docker" (`electricsql/electric` on Docker Hub). It needs a persistent filesystem for shape logs/metadata (`ELECTRIC_STORAGE_DIR`), an exposed HTTP port, and is "designed to run behind a caching proxy … or a CDN like Cloudflare or Fastly." Health checks at `/v1/health`. Deployable on AWS, GCP, DO, Fly.io, Render ([deployment guide](https://electric-sql.com/docs/guides/deployment)). Same conclusion as zero-cache regarding Laravel Cloud: no documented way to run a custom Docker service there, so it needs separate hosting — or the managed [Electric Cloud](https://electric-sql.com/docs/guides/deployment) ("simple, scalable, low-cost, managed Electric hosting").
- **Write path: explicitly not Electric's job.** "Electric does read-path sync… Electric does not do write-path sync. It doesn't provide (or prescribe) a built-in solution for getting data back into Postgres." The docs describe four patterns, all of which send writes through **your own API**: (1) plain online writes, (2) optimistic state (e.g. React `useOptimistic`), (3) shared persistent optimistic state, (4) through-the-database sync with an embedded local DB like PGlite ([writes guide](https://electric-sql.com/docs/guides/writes)). **This is the key architectural fit for Laravel:** your existing Laravel controllers/Form Requests/policies remain the entire write path unchanged; no push-protocol endpoint to implement, no mutator code in TypeScript required on the server.
- **Auth/permissions.** Electric's HTTP API "is public by default" and must be secured in production via an authorizing proxy, network rules, and/or an API secret; the documented patterns are **proxy auth** (route shape requests through your API, which validates the user and pins shape params server-side) and **gatekeeper auth** (your API issues shape-scoped tokens) ([security guide](https://electric-sql.com/docs/guides/security), [auth guide](https://electric-sql.com/docs/guides/auth)). A Laravel route can serve as that proxy/gatekeeper.
- **Neon compatibility: first-party on both sides.** Electric's docs have a dedicated Neon integration page (enable logical replication in Neon console, use the direct non-pooled connection string) ([Electric Neon page](https://electric-sql.com/docs/integrations/neon)), and Neon publishes its own Electric getting-started guide ([Neon guide](https://neon.com/guides/electric-sql)). Neon and Electric also co-developed PGlite ([Electric Neon page](https://electric-sql.com/docs/integrations/neon)). The same Neon caveats as Zero apply: logical replication is a permanent project-level switch, and an attached subscriber keeps compute from scaling to zero ([Neon logical replication docs](https://neon.com/docs/guides/logical-replication-neon), [scale-to-zero docs](https://neon.com/docs/guides/scale-to-zero-guide)).
- **Maturity/license.** GA since 1.0 on 2025-03-17: "the core Electric sync service APIs are now stable… no backwards-incompatible changes in patch or minor releases" ([1.0 blog post](https://electric-sql.com/blog/2025/03/17/electricsql-1.0-released)). Apache-2.0, open source at [electric-sql/electric](https://github.com/electric-sql/electric). (Note: the company's marketing site has since pivoted messaging toward "Electric Agents" built on the sync engine ([electricsql.com](https://electricsql.com/)); the sync-service docs remain at electric-sql.com.)
- **Zero vs Electric for a Laravel backend.** Zero gives a much richer client programming model (ZQL queries, optimistic mutators, automatic reconciliation) but its server contract (query endpoint returning ZQL ASTs, mutate endpoint speaking the push protocol) is TypeScript-first, with other languages possible but undocumented ([queries docs](https://zero.rocicorp.dev/docs/queries), [mutators docs](https://zero.rocicorp.dev/docs/mutators)). Electric is deliberately thinner — HTTP shape streaming for reads, bring-your-own writes — which composes cleanly with an existing Laravel API at the cost of hand-rolling optimistic state and merge/rollback logic on the client ([writes guide](https://electric-sql.com/docs/guides/writes)).

## 7. The local-files angle

**Neither Zero nor Electric addresses user-owned files on disk.** Both are Postgres sync engines: data originates in and is authoritative in the developer's Postgres database. Zero replicates Postgres → zero-cache's SQLite replica → a client-side store in IndexedDB ([llms.txt overview](https://zero.rocicorp.dev/llms.txt), [auth docs](https://zero.rocicorp.dev/docs/auth) — `zero.delete()` "deletes all data from the browser's IndexedDB database"). Electric streams Postgres data over HTTP into whatever client store you choose, including a local embedded PGlite database ([writes guide](https://electric-sql.com/docs/guides/writes)). In both cases the "local" copy is browser-managed storage, not user-visible files, and there is no free-tier story where data lives *only* on the user's machine — the upstream Postgres is required.

For a true Obsidian-like model (user-owned Markdown/JSON files in a folder they control), the browser building blocks are:

- **File System Access API** (`window.showOpenFilePicker` / `showDirectoryPicker` / `showSaveFilePicker`): lets a web page read and write real files/directories on the user's disk with permission prompts. `showDirectoryPicker` returns a `FileSystemDirectoryHandle` for enumerating and writing a folder — the closest browser primitive to a "vault." It is **not Baseline**: MDN marks it experimental and unsupported in some of the most widely-used browsers ([MDN: showDirectoryPicker](https://developer.mozilla.org/en-US/docs/Web/API/Window/showDirectoryPicker)); Chrome's docs say it's supported on most Chromium browsers, i.e. not Firefox/Safari ([Chrome developers](https://developer.chrome.com/docs/capabilities/web-apis/file-system-access)).
- **Origin Private File System (OPFS)** (`navigator.storage.getDirectory()`): fast, broadly supported (Baseline since March 2023 per MDN), but "private to the origin of the page and **not visible to the user** like the regular file system" ([MDN: OPFS](https://developer.mozilla.org/en-US/docs/Web/API/File_System_API/Origin_private_file_system)) — good for a local cache/database, not for user-owned vaults.
- **Desktop wrapper** (Electron/Tauri): the only path to full, cross-platform, user-visible file ownership plus background file watching, since browser pickers are Chromium-only and permission-gated. (General inference from the browser-support facts above; no single primary source.)

So "free tier = your files on your machine" is an application-architecture decision (file format + File System Access API or a desktop app), largely orthogonal to the Zero-vs-Electric choice, which only governs how *cloud-hosted* data syncs.

## 8. Provisioning Neon directly + Laravel Cloud app

- **Connecting Laravel to external Neon: supported and documented by Neon.** "Connecting to Neon from Laravel is the same as connecting to a standalone Postgres installation" — set `DB_CONNECTION=pgsql` and the Neon host/credentials in `.env` ([Neon Laravel guide](https://neon.com/docs/guides/laravel)). On Laravel Cloud you'd set these as environment variables instead of using an attached database; nothing in Laravel Cloud's docs prevents connecting to an external database (env vars are freely configurable — [environments API](https://cloud.laravel.com/docs/api/environments/add-environment-variables.md)), and Laravel Cloud publishes its outbound IP ranges for allowlisting on external services ([network docs](https://cloud.laravel.com/docs/network.md)). **Caveat:** no Laravel Cloud doc page explicitly demonstrates "bring your own external Postgres," so this is supported-by-mechanism rather than by an explicit guide.
- **Watch the SNI/endpoint-ID issue.** Older `libpq`/PDO_PGSQL builds fail with "The endpoint ID is not specified"; the fix is a newer client or passing `options=endpoint%3D<endpoint-id>` / `DB_PASSWORD=endpoint=<id>;<password>` ([Neon Laravel guide](https://neon.com/docs/guides/laravel), [Neon connection errors](https://neon.com/docs/connect/connection-errors)). Whether Laravel Cloud's PHP images carry a new-enough libpq is not documented — test early.
- **Pooling.** Neon provides PgBouncer in transaction mode (up to 10,000 client connections; pooled hostname = endpoint + `-pooler`). Use the pooled string for the Laravel app; use the **direct** string for migrations, `pg_dump`, and anything using logical replication (i.e. zero-cache/Electric upstream connections) ([Neon connection pooling docs](https://neon.com/docs/connect/connection-pooling)).
- **Autoscaling / scale-to-zero economics.** Neon computes autoscale and suspend after idle (default 5 minutes), but a connected logical-replication subscriber (zero-cache or Electric) keeps the compute active continuously ([Neon scale-to-zero docs](https://neon.com/docs/guides/scale-to-zero-guide)) — so adding a sync engine converts a scale-to-zero database into an always-on one. Also recall: inactive replication slots are dropped after ~40 hours and slot/sender limits are fixed at 10 ([Neon logical replication docs](https://neon.com/docs/guides/logical-replication-neon)).
- **Branching.** Neon branches are a headline feature (used for preview environments), but with Zero attached each active branch's compute can be kept running by its replication connection ([Zero docs, Neon branching note](https://zero.rocicorp.dev/docs/connecting-to-postgres)); Zero has a dedicated [previews doc](https://zero.rocicorp.dev/docs/previews) for the recommended approach.

## Recommendation-relevant summary

1. **Plain Laravel Cloud Postgres** is the only zero-extra-infrastructure option and the only one where Laravel Cloud's managed database can be used as-is.
2. **Zero on Laravel Cloud's database: not currently viable as documented** — no logical replication toggle, no Neon console access ([§2](#2-laravel-cloud-managed-postgres)). Zero effectively requires direct-provisioned Neon (or other Postgres) + zero-cache hosting elsewhere (or Cloud Zero at $30+/mo) + query/mutate endpoints, which are officially TS-first ([§3](#3-deploying-zero-cache)–[§4](#4-zeros-write-path-auth-and-permissions)).
3. **Electric has the same database blocker** but a lighter integration story for Laravel: reads via the Electric service (self-hosted elsewhere or Electric Cloud) proxied through Laravel routes for auth, and writes through the existing Laravel API with no new protocol ([§6](#6-electricsql-as-the-alternative)).
4. **Neither solves the Obsidian-style user-owned files vision** — that requires a file-format + File System Access API (Chromium-only) or a desktop wrapper, regardless of sync engine ([§7](#7-the-local-files-angle)).
5. Adding either sync engine to Neon **defeats scale-to-zero billing** because the replication connection keeps compute always on ([§1](#1-zeros-requirements-on-postgres), [§8](#8-provisioning-neon-directly--laravel-cloud-app)).
